# Vibe Hub · DESIGN HISTORY

这份文件用于在记忆丢失、目录漂移、版本分散时，保留 Vibe Hub 的设计连续性。

目标不是把每一句聊天都抄下来，而是把**真正影响设计方向的结论**写清楚。

---

## 当前结论（先看这里）

Vibe Hub 现在已经**不再继续做多方向分叉探索**，而是明确沿下面这个方向继续收：

### B 方向：云朵生命感版

关键词：
- 更柔软轮廓
- 更像呼吸的漂浮
- 更弱工具感
- 更像一团悬浮的语言云

这意味着后续设计原则应当是：
- 少一点“控制台 / 面板 / 工具箱”感
- 少一点为了比较而比较的多形态切换
- 多一点“语言被托起、被整理、再被送出”的生命感
- 让 Vibe Hub 更像一种**短暂凝结出来的语言载体**，而不是普通卡片组件

### V2 · 2026-03-24 · 当前稿前端原型（多形态比较阶段）
当前整理位置：
- `vibe-hub/versions/2026-03-24-v2-current-prototype`

来源：
- 原始目录为 `soul-chain/apps/vibe-hub`

这一版已确认落地的内容：
- 主卡片可拖动
- 三种外形可切换：
  - 透明信封形
  - 透明对话气泡形
  - 切角 Token 牌形
- 重写 / 压缩反馈更明显
- 背景更像真实桌面 / 壁纸
- 保留编辑、发送、改写、压缩、历史抽屉等交互
- 代码中已经预留 mock bridge 接口，便于后续接真实能力

这一版的设计阶段属性：
- 本质上属于**收集审美判断、比较不同外形方向**的阶段
- 重点不是最终定稿，而是为了判断：
  - 哪种更像系统浮层
  - 哪种更适合 Vibe Hub 这个名字
  - 哪种最不像普通圆角面板
  - 卡片在不同桌面位置下是否还成立

设计意义：
- 它把 Vibe Hub 从“能用”推进到了“开始有明显视觉性格”
- 但它仍然偏**探索版**，没有完成最终收口

### V3 · 2026-03-24 · 云朵生命感版（收口尝试稿）
当前整理位置：
- `vibe-hub/versions/2026-03-24-v3-cloud-living`

这一版的明确目标：
- 沿 B 方向继续，不再分叉
- 去掉多外形切换
- 从“形态比较”进入“单方向打磨”

这一版已做的调整：
- 去掉三种外形切换
- 主卡片改成更柔软、不规则的云团轮廓
- 漂浮感改得更轻，强调呼吸和悬浮
- 背景从桌面装饰感，往天空 / 雾团 / 轻浮层气质移动
- 局部文案从工具型表达改成更偏“语言云”表达
- 仍保留核心交互：
  - 拖动
  - 编辑
  - 重写
  - 压缩
  - 发送
  - Token Stack 历史抽屉

已知修复：
- 这版曾出现“卡片不在视口中央、看不见”的问题
- 原因是拖动位置记忆和动画 transform 叠加
- 后来已修成：
  - 默认强制居中
  - 更换 localStorage key，避免继承旧错误位置
  - 去掉会影响定位的 transform 位移动画，只保留阴影呼吸感

后续判断：
- 用户对这版的核心不满很明确：
  - 漂浮感一度丢了
  - 左上角信息太多
  - 文案过虚
  - 整体“既不云朵，也不透明”
- 也就是说，V3 虽然完成了“收口到 B 方向”的动作，但并没有成为最终可接受的视觉答案。

设计意义：
- 它的重要性主要在于：
  **把发散阶段停住，并逼出了后续真正的判断标准**

- 它的重要性不只在视觉，而在于：帮助用户理解“我从哪里输入、卡片经过什么加工、最后发到哪里”

## 稳定不变的产品理解

下面这些不是某一版的偶然表现，而是 Vibe Hub 持续成立的核心理解：

### 1. Vibe Hub 不是普通聊天输入框
它不是为了替代短指令直发，而是为了接住：
- 更长的描述
- 更复杂的任务意图
- 需要机器整理后再确认的内容

### 2. 应该存在两条路径
始终要保留这两个语义路径：
- **短指令**：一句话直接发
- **长描述**：先进 Vibe Hub，整理后再发

UI 上最好让这两条路清楚，而不是把一切都塞进同一个输入框里。

### 3. Vibe Hub 是“整理与确认”的载体
它的价值不只是展示文字，而是让用户感受到：
- 机器已经整理过这段意图
- 但人仍然保有最后修改权
- 然后再把它送去某个 agent / 调度器 / 当前应用

### 4. Token Stack 很重要
历史抽屉不是附属品，它是体验的一部分：
- 用户需要回看自己送过什么
- 用户需要重新发送、编辑后再发、转发给其他目标
- 它让 Vibe Hub 不只是一次性输入，而是一个可回溯的语言工作流

### 5. 后端真实接线还没完成
现阶段主要仍是前端原型与交互原型。
目前代码里有 mock bridge，但还没有真正接到完整真实能力。

## 后续继续设计时的优先原则

如果未来再次从这里继续，不要重新大范围发散，优先按下面顺序推进：

### Priority 1 · 把 B 方向做对
先判断当前云朵生命感版是否真正做到：
- 不是普通玻璃卡片
- 不像仪表盘控件
- 有漂浮生命感
- 有“语言被托起”的感觉

### Priority 2 · 保住交互清晰度
即使变软，也不能牺牲这些基本清晰度：
- 现在要发给谁
- 当前正文是什么
- 可以重写 / 压缩 / 发送
- 历史卡片怎么回看和重发

### Priority 3 · 再继续弱化工具感
可以继续削弱：
- 过强的按钮感
- 过硬的边框感
- 过像软件控制台的块状结构

但不要把它做得梦幻到不可操作。

### Priority 4 · 等视觉方向稳定后，再接真实能力
等设计方向更稳，再考虑把 mock bridge 接到真实链路：
- OpenClaw bridge
- 调度器
- companion app 原生壳层
- 本地数据库 / 文件存储 / dispatch service
