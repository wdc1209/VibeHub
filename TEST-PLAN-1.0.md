# Vibe Hub 1.0 发布前测试清单

更新时间：2026-04-04

## 发布策略

Vibe Hub 1.0 采用“固定支持名单型”。

原则：
- 只展示预置支持名单中，当前机器上已扫描到的输出点
- 不展示未安装的软件
- 只有逐项测试通过的输出点，才能进入 1.0 正式支持清单

## 当前 1.0 候选清单

### 输出终端

- 剪贴板
- Codex
- AntiGravity
- OpenClaw Web Chat

### 输出网站

- Google Chrome

## 第二阶段候选

以下项暂列为候选，不直接进入 1.0 正式支持清单：

- ChatGPT
- Cursor
- Discord
- Notion
- Grok
- Twitter / X
- Zhihu
- Xiaohongshu
- YouTube

## 测试通过标准

每个准备进入 1.0 的输出点，至少需要通过以下项目：

1. 启动扫描后能正确显示在面板
2. 状态文案正确
3. 可连接项点击后能正确连接
4. 窗口枚举正确
5. 发送成功
6. 发送后上一轮内容被清空
7. 失败时有清晰错误提示

## 阶段一：App 类优先测试

### 1. 剪贴板

- [ ] 面板显示正常
- [ ] 发送后系统剪贴板内容正确
- [ ] 发送后正文被清空
- [ ] 再次轮询不会把旧内容刷回

### 2. Codex

- [ ] 检测到已安装
- [ ] 状态显示“已连接”或“可连接”
- [ ] 点击可连接后能接入
- [ ] 窗口列表可选
- [ ] 发送成功
- [ ] 发送后正文被清空
- [ ] Busy / queue 状态提示正确

### 3. AntiGravity

- [ ] 检测到已安装
- [ ] 状态显示正确
- [ ] 点击可连接后能接入
- [ ] 窗口列表可选
- [ ] 发送成功
- [ ] 发送后正文被清空

### 4. OpenClaw Web Chat

- [ ] 面板显示正常
- [ ] 主会话窗口列表正常
- [ ] 固定微信会话显示正常
- [ ] 发送到普通 main 会话成功
- [ ] 发送到微信会话成功
- [ ] 发送后正文被清空
- [ ] 轮询不会把刚发完的内容刷回

## 阶段二：网站类测试

### 5. Google Chrome

- [ ] Browser Bridge 扩展已连接
- [ ] 面板归类在“输出网站”
- [ ] 状态文案准确
- [ ] 页面 / 窗口可正确枚举
- [ ] 若页面发送未完成，需在发布说明中明确写出当前能力边界

## 发布结论门槛

### 可直接进入 1.0 的条件

- 输出终端全部通过
- 输出网站至少对当前纳入 1.0 的项全部通过
- 语音输入、整理、发送、微缩窗口、完整窗口均完成回归

### 不应进入 1.0 的情况

- 面板中出现未测过的终端
- 某终端可显示但连接链路不稳定
- 某终端发送后无法清空
- 某终端的状态文案与真实能力不一致
