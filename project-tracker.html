设计原则

  • Vibe Hub 不是普通聊天输入框，而是承接长描述、整理意图、再确认送出的前台载体。
  • 它必须是一个真正独立、可自由拖动的 macOS 窗口，而不是网页里模拟出来的假桌面卡片。
  • 输入与输出边界要清晰：输入来自 OpenClaw + bridge，输出通过 bridge + opencli / 目标适配器完成。
  • 状态面板里的每一项都必须有真实来源，不允许装饰性数字或虚构状态。
  • Vibe Hub 的价值在于“整理与确认”，人始终保留最后修改权。
  • 历史记录不是附属品，而是可回看、可筛选、可重发的语言工作流。

项目目标

  • 做成真正原生的 SwiftUI / AppKit macOS utility window。
  • 继承 V6 / v8 的产品结构：主编辑区、原始输入、状态面板、历史面板、发送入口。
  • 读 bridge 的真实状态与 token session。
  • 通过 bridge 将整理后的内容真实发送到 Codex / 目标终端。
  • 保留本地历史记录、筛选与回看能力。
  • 把安装流程收束到可重复的本地安装路径，再推进对外分发。

职责边界

Vibe Hub 自己负责
  • 窗口 UI
  • 内容编辑
  • 状态展示
  • 本地历史记录
  • 发送入口
不由 Vibe Hub 负责
  • 直接从聊天软件表面抓输入
  • 直接负责目标终端投递
  • 替代 OpenClaw / bridge 的路由职责
