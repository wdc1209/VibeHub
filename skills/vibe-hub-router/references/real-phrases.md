# Vibe Hub Router Real Phrases

这份文件不是规则定义，而是产品口径校准样例。

用途：

- 对齐“哪些话应该先进 Vibe Hub”
- 对齐“哪些话仍然留在当前聊天里直接处理”
- 对齐“哪些话不能猜，需要先澄清”

## `direct` · 当前聊天直接处理

这些话虽然可能和工作有关，但主目标是解释、判断、分析、建议，而不是马上形成外发稿。

| 用户原话 | 预期 | 为什么 |
| --- | --- | --- |
| `解释一下这个报错` | `direct` | 明确是解释请求 |
| `帮我分析这段代码为什么慢` | `direct` | 当前会话内分析 |
| `这个架构合理吗` | `direct` | 讨论，不是交付 |
| `你觉得这段话会不会太强硬` | `direct` | 咨询判断，不是外发执行 |
| `review 一下这个 PR 风险` | `direct` | 代码审查，不是外发稿 |
| `继续` | `direct` | 短控制指令 |
| `暂停` | `direct` | 短控制指令 |
| `发送` | `direct` | 会话内控制动作，不等于新建卡片 |

## `vibe-hub` · 必须先进 Vibe Hub

这些话已经带有“整理后交付出去”的心智，不应该让 OpenClaw 直接吞掉。

| 用户原话 | 预期 | 为什么 |
| --- | --- | --- |
| `把这段整理后发给 Codex` | `vibe-hub` | 明确外发给 agent |
| `给客户写一版回复` | `vibe-hub` | 面向外部对象的可交付稿 |
| `帮我整理成发给老板的周报口径` | `vibe-hub` | 对外发送前整理 |
| `下面这段需求你整理成可以交给 Cursor 的执行任务` | `vibe-hub` | 明确交付给外部执行者 |
| `这是我要发在群里的说明，你先整理一下` | `vibe-hub` | 面向外部渠道 |
| `把这三点压成一版可以直接发邮件的内容` | `vibe-hub` | 明确交付形态 |
| `请把下面内容变成可发送的 Codex 指令卡` | `vibe-hub` | 明确是外发前卡片 |
| `给同事起草一条同步消息` | `vibe-hub` | 对外对象 + 消息草稿 |

## `clarify` · 不能猜，先问一句

这些话有外发倾向，但目标或边界还不够明确。

| 用户原话 | 预期 | 为什么 |
| --- | --- | --- |
| `把这个整理一下然后发出去` | `clarify` | 要发，但没说发到哪里 |
| `你帮我整理一版，我等下拿去用` | `clarify` | 可能是自用，也可能是外发 |
| `做个能发的版本` | `clarify` | 有交付语气，但缺收件方 |
| `帮我出一版，我要对外` | `clarify` | 边界还不够具体 |

推荐澄清句：

- `这是要我现在直接在聊天里处理，还是先放进 Vibe Hub 整理后再发送？`

## Active Token Session · 已进卡后的追加输入

这里的重点不是句子本身，而是它们出现在“当前已经有 Vibe Hub session”时的含义。

| 用户原话 | 预期 | 为什么 |
| --- | --- | --- |
| `补充一点：把预算风险也写进去` | `vibe-hub` | 明确是对当前卡片追加 |
| `再加一条，强调这周必须上线` | `vibe-hub` | 生命周期内补充 |
| `把语气再收一点` | `vibe-hub` | 当前卡片编辑要求 |
| `顺便写上时间节点` | `vibe-hub` | 当前卡片编辑要求 |
| `另外补一句：先不要承诺交期` | `vibe-hub` | 当前卡片编辑要求 |
| `取消` | `direct` | 控制动作，不是继续并稿 |
| `发送` | `direct` | 控制动作，应交给当前 UI/会话行为 |

## 边界提醒

- `给我想想怎么回复客户比较稳` 更像 `direct`，因为用户先要建议，不一定已经进入“形成外发稿”的阶段。
- `给客户写一版回复` 更像 `vibe-hub`，因为用户已经要求产出可交付文本。
- `把下面这段代码发给 Codex 看看` 即使内容里是代码，也更像 `vibe-hub`，因为关键不在代码，而在“准备外发给 Codex”。
